## Session: SMS Online Pro Documentation & Service Viewer

### CORRECTION MAJEURE — Analyse API
**J'avais totalement tort sur le protocole API de SMS Online Pro.** Mon analyse initiale du bundle React (REST/JSON avec JWT Bearer) était trompeuse — ce bundle est le **frontend SPA**, pas l'API réelle.

**La vérité :** SMS Online Pro utilise EXACTEMENT le même protocole sms-activate que Grizzly SMS — `handler_api.php?api_key=X&action=Y`. C'est compatible à 95%.

### Comment j'ai trouvé la vraie doc
1. Le bundle React (`index-DdifeFfq.js`) contient un routeur qui montre que `/docsPage` charge une iframe vers `/docs/api/en/index.html`
2. Cette page est du HTML statique hébergé à `https://sms-online.pro/docs/api/en/index.html`
3. Les pages de documentation individuelles sont à `/docs/api/en/number.html`, `/docs/api/en/get_prices.html`, etc.
4. Les listes de services/pays sont dans un fichier JS à `/docs/api/js/script.js`

### API endpoints confirmés
Base: `https://sms-online.pro/stubs/handler_api.php` (GET/POST, api_key)

1. `getNumber` — Demander numéro
2. `setStatus` — Changer statut (1:prêt, 3:nouveau code, 6:compléter, 8:annuler)
3. `getStatus` — Voir statut
4. `getBalance` — Solde
5. `getNumbersStatus` — Quantité dispo par service
6. `getPrices` — Prix par pays/service (JSON)
7. `getOperators` — Opérateurs par pays (JSON)
8. `getTopCountriesByService` — Top pays
9. Rent API complète (location de numéros)

### Conséquences sur le plan de migration
- **Le format API étant le même que Grizzly**, la migration est bcp plus simple qu'imaginé
- Pas besoin de rewrite du client HTTP — juste changer l'URL de base et les codes services/pays
- Les statuts d'activation (1,3,6,8) sont les mêmes
- Les réponses (ACCESS_NUMBER, STATUS_OK, etc.) sont les mêmes
- Points à mapper : codes services, codes pays, structure des prix
